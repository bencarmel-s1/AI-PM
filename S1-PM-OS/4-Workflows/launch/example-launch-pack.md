## Artifact 1 — Release Notes

> **Where this goes:** Confluence (Product Updates space) + linked to Jira epic PULSE-388

### Pulse Alerts 2.0: Anomaly Detection — Release Notes
**Version:** v3.8.0
**Release date:** 2026-03-17
**Platform area:** Pulse Alerts

#### What's new

| Capability | Description | Available to |
|---|---|---|
| Automatic anomaly detection | Pulse now automatically detects unusual spikes and drops in your key metrics — no manual threshold setup required | Growth, Professional, Enterprise |
| Root cause summary | Each anomaly alert includes a 2-hour correlation window showing what changed before the anomaly occurred | Growth, Professional, Enterprise |
| Multi-metric correlation | If multiple metrics shifted at the same time, the alert groups them and surfaces the likely shared cause | Growth, Professional, Enterprise |
| iOS push notifications | Anomaly alerts delivered to your phone via the Pulse iOS app | Growth, Professional, Enterprise |
| Slack thread digest | Anomaly alerts are delivered as Slack threads with one-click drill-down to the Pulse dashboard | Growth, Professional, Enterprise |

#### How it works
Pulse Alerts 2.0 uses a statistical baseline model trained on each account's last 90 days of data. The model detects deviations that fall outside the expected range for that day/hour pattern (accounting for seasonality), and fires an alert when confidence is ≥ 95%. Root cause context is generated by correlating the anomaly time window against all tracked events in the same period.

#### What changed in the UI
- New **Anomaly Detection** section in the Alerts dashboard (replaces the old "Smart Alerts" beta label)
- Alert detail view now includes a **Root Cause** card showing correlated events
- Alert card shows **Related Alerts** if multiple metrics fired within the same 2-hour window
- **Settings → Alerts → Anomaly Detection** — admin panel for enabling/disabling detection per metric and adjusting sensitivity (Low / Medium / High)

#### API / integration notes
- New webhook event: `alert.anomaly_detected` — includes anomaly details, metric name, deviation %, and root cause correlation data
- Updated event: `alert.triggered` now includes `anomaly_context` object for anomaly-type alerts
- Existing threshold-based alert webhooks are unchanged

#### Migration / enablement
Anomaly Detection is **enabled by default** for all accounts on Growth tier and above. Detection starts immediately and builds a baseline over the first 7 days. Customers may see a higher alert volume in the first week as the model calibrates — this is expected and will stabilize. Admins can reduce sensitivity in **Settings → Alerts → Anomaly Detection**.

#### Known limitations
- Anomaly detection requires ≥ 7 days of historical data per metric before alerts become accurate. New accounts or recently added metrics will see a "Calibrating" state during this period
- Android push notifications are not available in v1 — targeted for Q2
- Detection covers standard Pulse metrics only. Custom event metrics are not supported in v1 — targeted for Q3
- Alert snooze and acknowledgment workflow not available in v1 — targeted for v1.1

### Pulse Alerts 2.0: Anomaly Detection — Sales Enablement Brief

**What's new in one sentence:**
Pulse now automatically detects when something important changes in your product metrics — before your customers even notice — and tells you why.

**The customer problem we're solving:**
Product and growth teams are flying blind between analysis sessions. They manually check dashboards every morning, set up brittle threshold alerts that fire constantly (or not at all), and find out about real problems from customers or support tickets — not from their own analytics tool. The result is slow incident response, missed growth opportunities, and low trust in the alerting system.

**What they can do now:**
- Stop configuring thresholds — Pulse learns what "normal" looks like for each account and alerts automatically
- Understand *why* something changed, not just *that* it changed — root cause context surfaces in the alert
- Get alerted where they already work — Slack threads + iOS push, not just email
- Act on correlated signals — if three metrics dropped at the same time, they're grouped in one alert

**Who to lead with:**

| Persona | Why they care | Opening line |
|---|---|---|
| VP Product / CPO | Miss fewer critical product events; stop finding out from customers | "When was the last time you found a product problem from your analytics tool before a customer reported it?" |
| Product Manager | Stop babysitting dashboards; get smart alerts that tell you what's wrong, not just what changed | "Pulse now monitors your metrics 24/7 and alerts you with context — not just a number that moved." |
| Growth Lead | Catch conversion drops faster; correlate anomalies to recent releases or experiments | "The moment your activation funnel drops, Pulse tells you what changed in the 2 hours before it happened." |
| Data Analyst | Fewer "why did this spike?" Slack questions from PMs | "Anomaly context reduces the first-response investigation time from 30 minutes to about 5." |

**Competitive angle:**

| Competitor | Their approach | Our advantage |
|---|---|---|
| Amplitude | Threshold-based alerts; no automatic anomaly detection without premium add-on | Pulse anomaly detection is included in Growth tier — no add-on pricing required |
| Mixpanel | Alerts available but no root cause context; customers must investigate manually | Root cause correlation is built into the alert — no separate investigation step |
| Heap | No native alerting; relies on third-party integrations | Pulse delivers alerts natively in Slack and iOS — no Zapier required |

**Objections and responses:**

*"We already have threshold alerts set up in Mixpanel / Amplitude..."*
> Threshold alerts require you to know what threshold to set — and they require maintenance every time your baseline shifts (seasonal traffic, new feature launches). Pulse anomaly detection adapts to your baseline automatically. It alerts on what's abnormal for *you*, not on an arbitrary number you set six months ago.

*"Won't this alert too much and become noise?"*
> The model requires 95% confidence before firing. In the first 7 days it calibrates — you may see slightly higher volume during that window. After calibration, alert volume is typically lower than manually configured threshold systems. Admins can also adjust sensitivity (Low / Medium / High) per metric.

*"Is anomaly detection included in our plan or is it an add-on?"*
> Included in Growth, Professional, and Enterprise at no additional charge. Confirm current plan with your AE.

**Demo path:**
1. Open the Alerts dashboard → show the new Anomaly Detection section
2. Click into a recent anomaly alert → walk through the Root Cause card (correlated events)
3. Show the Slack thread format — one-click drill-down to the dashboard
4. Show Settings → Anomaly Detection → sensitivity slider

## Artifact 3 — CS & Support Brief

> **Where this goes:** CS Confluence space (`CS Hub → Product Updates`) + `#cs-product-updates` Slack channel

### Pulse Alerts 2.0: Anomaly Detection — CS & Support Brief

**What shipped:**
Pulse now automatically detects anomalies in customer metrics and delivers alerts with root cause context, without requiring manual threshold configuration. Available on Growth, Professional, and Enterprise tiers. Enabled by default.

**Which customers have access:**
All accounts on Growth tier and above. Detection is **enabled by default** — customers don't need to do anything to turn it on, but admins can adjust sensitivity or disable specific metrics in Settings.

**How customers will find it:**
- The Alerts dashboard now has an "Anomaly Detection" section (visible immediately)
- Customers who already use Slack alerts will see the new alert format in existing connected channels
- iOS app users will receive push notifications if they have the app installed

**Top questions customers will ask:**

*Q: I'm getting more alerts than before — is something wrong?*
A: This is expected in the first 7 days while the model calibrates to each account's baseline. Alert volume should stabilize after the calibration period. If it hasn't stabilized after 10 days, ask them to check Settings → Anomaly Detection and reduce sensitivity. Escalate to product if volume remains high after 14 days.

*Q: How does it know what's "normal"? Can I see the baseline?*
A: The model uses 90 days of historical data and accounts for day-of-week and time-of-day patterns. We don't currently surface the baseline in the UI — it's on the roadmap. Let customers know that confidence must reach 95% before an alert fires.

*Q: The alert says there's an anomaly but I don't see anything wrong in my dashboard.*
A: This can happen when the anomaly is a small deviation from a narrow baseline. Recommend the customer adjust sensitivity to "Low" in Settings if their metrics are naturally variable. Log these cases — if we see this pattern across multiple accounts, it's a product signal.

*Q: Can I snooze or acknowledge alerts?*
A: Not yet — this is the most common request we're hearing and it's targeted for v1.1 (Sprint 49, approximately 4 weeks out). For now, customers can disable specific metrics in Settings as a workaround.

*Q: Android push notifications aren't working.*
A: Android push is not available in v1 — iOS only. Android is targeted for Q2. Communicate this timeline to the customer if they ask.

**Escalation path:**
- Slack: `#alerts-support` — PM and eng team monitor this channel
- Jira: Project `PULSE`, component `Anomaly Detection`
- For P0 issues (false alert storms, alerts not firing): page on-call

**Known issues to communicate proactively:**
- 7-day calibration period: customers may see more alerts than expected in the first week. This is expected.
- Android push: not available in v1; Q2 timeline.
- Alert snooze: not available in v1; v1.1 timeline (Sprint 49).
